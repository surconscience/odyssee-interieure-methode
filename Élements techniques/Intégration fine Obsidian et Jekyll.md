# Intégrer Obsidian et Jekyll sans dupliquer le code

L’utilisation conjointe d’**Obsidian** (avec le plugin Dataview) et d’un site **Jekyll** peut entraîner des doublons de code pour afficher des contenus dynamiques (listes, tableaux de fiches, etc.). Voici des pistes pour mieux intégrer les deux environnements : plugins Obsidian facilitant l’export vers Jekyll, méthodes pour centraliser les données (une _single source of truth_), techniques d’inclusion de blocs communs, alternatives aux approches Dataview/Liquid, et enfin un tour d’horizon d’autres solutions de publication (Quartz, Obsidian Publish, thèmes Jekyll adaptés).

## Plugins Obsidian pour une meilleure intégration avec Jekyll

Plusieurs plugins **communautaires Obsidian** peuvent aider à exporter ou synchroniser le contenu vers Jekyll (HTML/Liquid) en minimisant les doublons :

- **Bulk Exporter** : Un plugin exploitant le langage de requête de Dataview pour **exporter des notes** de manière structurée. Il permet de filtrer et organiser des notes par métadonnées, puis de les exporter dans une arborescence adaptée à un générateur de site statique (**Jekyll**, Hugo...). Les liens internes sont réécrits automatiquement. Ce plugin est _idéal pour intégrer une volumineuse collection de notes dans Jekyll_, en évitant les manipulations manuelles[obsidianstats.com](https://www.obsidianstats.com/plugins/bulk-exporter#:~:text=The%20Bulk%20Exporter%20plugin%20allows,The%20plugin).
    
- **Dataview Publisher** : Ce plugin étend Dataview en **convertissant les résultats de requêtes en Markdown statique**, inséré directement dans vos notes[obsidianstats.com](https://www.obsidianstats.com/plugins/dataview-publisher#:~:text=The%20Dataview%20Publisher%20plugin%20extends,it%20more%20portable%20and%20accessible). Concrètement, on encadre une requête Dataview d’un marqueur `%% DATAVIEW_PUBLISHER` et le plugin y injecte le résultat sous forme de texte Markdown. Les données dynamiques deviennent donc statiques et **compatibles Obsidian Publish et Jekyll** (puisque le Markdown généré sera rendu comme du contenu normal)[obsidianstats.com](https://www.obsidianstats.com/plugins/dataview-publisher#:~:text=The%20Dataview%20Publisher%20plugin%20extends,it%20more%20portable%20and%20accessible). Par exemple, une requête Dataview `LIST FROM #tag` sera remplacée par une liste de liens vers les notes correspondantes[github.com](https://github.com/udus122/dataview-publisher#readme#:~:text=,DATAVIEW_PUBLISHER%3A%20end). _Avantage :_ plus besoin d’écrire une boucle Liquid équivalente côté Jekyll, le contenu est déjà préparé dans la note.
    
- **Dataview Serializer** (alternative similaire) : Génère aussi du Markdown statique à partir de Dataview, avec l’objectif d’être **compatible Obsidian Publish** et de faire apparaître les liens dans la vue Graphe[obsidianstats.com](https://www.obsidianstats.com/plugins/bulk-exporter#:~:text=Dubois%20Obsidian%20plugin%20that%20updates,a%20year%20ago%20by%20alexeiskachykhin). C’est une autre option pour « figer » les résultats des requêtes.
    
- **Digital Garden (plugin)** : Plugin de publication en ligne d’Ole Eskild qui offre une intégration poussée : il prend en charge les **backlinks**, la navigation dans l’arborescence, les médias embarqués, et même la visualisation de données via des requêtes Dataview[obsidianstats.com](https://www.obsidianstats.com/plugins/digitalgarden#:~:text=The%20Digital%20Garden%20plugin%20allows,for%20a%20unique%20web%20presence). Ce plugin automatise la publication sur GitHub/Vercel et offre un _workflow_ fluide pour maintenir un **digital garden**. _Limite :_ il ne s’appuie pas sur Jekyll, mais sur un site Next.js, donc c’est plutôt une alternative qu’une intégration directe de Jekyll. Il est néanmoins compatible avec les fichiers Obsidian (wikliens, transclusions) et très populaire (plus de 60 000 téléchargements)[obsidianstats.com](https://www.obsidianstats.com/plugins/digitalgarden#:~:text=1854).
    
- **Obsidian Git** (à considérer) : Ce plugin permet de synchroniser votre voûte Obsidian avec un dépôt GitHub. Couplé à GitHub Pages ou une action CI/CD Jekyll, il facilite l’**export automatisé** des notes vers le site Jekyll à chaque _commit_. Il ne règle pas la duplication de code en soi, mais assure que vos contenus Obsidian (après conversion éventuelle des liens ou requêtes) sont publiés sans effort manuel.
    
- **Autres plugins utiles** : _Meld Build_ (exécuter du JS pour rendre des modèles et requêtes Dataview en notes statiques), _Templater_ (scripting de modèles, voir plus bas pour centraliser les requêtes), _Metadata Extractor_ (exporter les métadonnées YAML des notes pour usage externe), etc. Ces outils peuvent compléter votre flux de travail selon vos besoins spécifiques.
    

## Centraliser une source de vérité pour les données (YAML/Markdown)

Pour éviter d’écrire la même logique de filtrage ou le même jeu de données à deux endroits (Dataview vs. Liquid), il est recommandé de **centraliser vos données** dans vos fichiers Markdown via les _métadonnées YAML_ ou un fichier source unique, afin que Jekyll et Obsidian les exploitent tous deux :

- **Utiliser le YAML front matter des notes comme base de données** : Obsidian (Dataview) et Jekyll partagent la capacité de lire les métadonnées en en-tête YAML des fichiers Markdown. En structurant bien ces métadonnées (par exemple un type de note `type: Fiche` et des champs comme `titre`, `auteur`, etc.), on peut ensuite :
    
    - Côté Obsidian, écrire une requête Dataview comme `TABLE titre, auteur FROM "" WHERE type="Fiche"` pour lister toutes les fiches avec leurs champs, triées comme voulu.
        
    - Côté Jekyll, utiliser une collection ou une boucle Liquid sur `site.pages` (ou `site.fiches` si vous configurez une collection) pour parcourir les notes de type « Fiche » et générer le tableau HTML correspondant (en injectant les mêmes champs `titre`, `auteur` stockés dans le YAML). **Ainsi, les données (le contenu des champs)** ne sont définies qu’une fois dans chaque note, et chaque système les met en forme à sa manière sans dupliquer les valeurs. Seule la _logique de requête/affichage_ diffère, mais vous évitez la duplication de contenu.
        
- **Stocker des données partagées dans un fichier unique** : Si vous avez un ensemble de données tabulaires ou une liste maintenue indépendamment des notes (par ex., un registre ou un lexique), envisagez de le conserver dans un format utilisable par Jekyll **et** Obsidian. Par exemple, un fichier YAML dans le dossier `_data` de Jekyll (ex: `donnees.yaml`) sera accessible via `site.data.donnees` en Liquid. Pour Obsidian, vous pourriez tenir une note Markdown (`donnees.md`) contenant cette liste (sous forme de tableau Markdown ou de liste à puces) – ou même intégrer le YAML entre balises de code – pour pouvoir la consulter et éventuellement la traiter. Certes, Dataview ne lit pas directement un fichier YAML externe, mais vous pourriez écrire un script (Templater + DataviewJS) pour charger ce fichier et afficher son contenu. L’important est que la **source de vérité soit unique** : par exemple, vous modifiez le fichier YAML, puis vous lancez un petit script dans Obsidian qui lit ce YAML et met à jour une note de présentation. Cela évite de ressaisir les mêmes informations deux fois.
    
- **Éviter la logique complexe en double** : Si une certaine logique de calcul est nécessaire (ex: filtrer les fiches selon une date ou un statut), essayez de la limiter à un seul endroit. Soit vous la gérez entièrement via Dataview et vous exportez le résultat statiquement (avec Dataview Publisher ou Templater), _soit_ vous la gérez via Liquid côté Jekyll. Avoir deux systèmes différents appliquer des critères de filtrage identiques augmente le risque d’incohérence. Par exemple, pour un tri des « notes récentes », vous pouvez choisir de le faire uniquement via Obsidian (Dataview script qui met à jour une note "Dernières notes" avant publication[joschua.io](https://joschua.io/posts/2023/09/01/obsidian-publish-dataview/#:~:text=Outputting%20Markdown%20from%20Dataview%20Queries)[joschua.io](https://joschua.io/posts/2023/09/01/obsidian-publish-dataview/#:~:text=%2F%2F%20Update%20Publish%20Files.md%20%3C,mtime%20desc%20LIMIT%207)), ou uniquement via Jekyll (Liquid peut trier par date de modification si vous passez cette info en front matter). L’important est de **choisir un maître d’œuvre** pour ces données dynamiques.
    

En résumé, exploitez au maximum les **métadonnées partagées** entre Obsidian et Jekyll : le YAML front matter est le point commun évident[forum.obsidian.md](https://forum.obsidian.md/t/jekyll-integration/8544#:~:text=,be%20preserved%20with%20much%20attention). De même, structurez vos notes de façon cohérente (titres, tags) pour pouvoir les sélectionner facilement dans les deux contextes.

## Blocs de contenu communs sans duplication (transclusion & includes)

Une problématique classique est de vouloir inclure un même **bloc de contenu** (par ex. un tableau de fiches, une section de texte standard) dans _plusieurs pages_ sans le dupliquer. Obsidian et Jekyll ont chacun leurs mécanismes d’inclusion, qu’on peut combiner astucieusement :

- **Transclusions Obsidian** : Obsidian permet nativement d’_embarquer_ le contenu d’une note A dans une note B via la syntaxe `![[Nom de la note A]]`. Ce _transclude_ affiche en prévisualisation tout le contenu de la note A à l’endroit désiré dans la note B. C’est très utile pour réutiliser, par exemple, un paragraphe standard, un tableau, ou une fiche entière sans faire de copier-coller. _(Astuce :_ on peut même transclure une section spécifique d’une note via `![[Note A#Section]]`).
    
- **Includes Jekyll** : Côté Jekyll, on peut réaliser la même chose grâce aux **includes** Liquid. En plaçant un fichier snippet (Markdown ou HTML) dans le dossier `_includes` de Jekyll, on peut l’insérer dans n’importe quelle page via `{% include monSnippet.md %}`. Jekyll va alors rendre le contenu du snippet **comme partie de la page**. Si on ne veut pas utiliser `_includes`, il existe aussi `{% include_relative %}` qui permet d’inclure un fichier Markdown situé dans le même dossier que la page courante[stackoverflow.com](https://stackoverflow.com/questions/41966638/jekyll-on-github-pages-include-markdown-in-another-markdown-file#:~:text=,md%20and%20the%20live%20page). Dans tous les cas, l’inclusion est traitée au moment de la génération du site, ce qui évite d’avoir des duplications de texte.
    

**Combiner les deux approches** : l’idée est de maintenir vos blocs communs dans des fichiers séparés, accessibles à la fois par Obsidian et Jekyll. Par exemple, si vous avez un tableau de fiches que vous voulez voir apparaître sur plusieurs pages :

- Créez une note Obsidian (ex : `snippet_tableau_fiches.md`) contenant ce tableau (ou même une requête Dataview Publisher qui génère ce tableau statiquement).
    
- Dans Obsidian, à chaque endroit où le tableau doit apparaître, insérez `![[snippet_tableau_fiches]]`. Vous verrez ainsi toujours la version à jour de ce tableau.
    
- Dans Jekyll, placez ce même fichier dans `_includes` (ou utilisez include_relative) et insérez-le via `{% include snippet_tableau_fiches.md %}` là où nécessaire. Sur le site, Jekyll collera le HTML du tableau à ces emplacements, **sans** que vous ayez à dupliquer le code du tableau dans chaque page.
    

De cette manière, vous modifiez une seule fois le contenu dans `snippet_tableau_fiches.md`, et _toutes_ les pages Obsidian ou Jekyll qui l’incluent se mettent à jour. Cela élimine la redondance. Cette technique marche pour n’importe quel bloc réutilisable : encadrés d’astuce, sections de biographie, définitions, etc. Elle capitalise sur le fait que **vos notes Markdown sont la source commune** : Obsidian les voit directement, et Jekyll via son système d’includes peut les incorporer au build final[stackoverflow.com](https://stackoverflow.com/questions/41966638/jekyll-on-github-pages-include-markdown-in-another-markdown-file#:~:text=,md%20and%20the%20live%20page).

_Remarque :_ Obsidian ignore les balises Liquid (`{% %}`) si vous les laissez dans vos notes – elles apparaîtront simplement comme du texte brut en prévisualisation. Pour éviter cela, vous pouvez conserver les includes Jekyll seulement dans une version d’export (par exemple en insérant un tag spécial que vous remplacez lors du build). Mais l’approche la plus simple est celle décrite ci-dessus : utiliser des fichiers de snippet communs et la transclusion côté Obsidian, plutôt que de mêler du Liquid dans vos notes.

## Alternatives utilisables dans les deux contextes (Dataview vs Liquid)

Si l’on cherche des _solutions unifiées_ qui pourraient remplacer à la fois Dataview et Liquid pour certaines tâches, il faut reconnaître qu’il n’existe pas de langage ou d’outil _unique_ s’exécutant _nativement_ dans Obsidian _et_ Jekyll. Toutefois, voici quelques alternatives et astuces pour minimiser les divergences entre les deux mondes :

- **Générer du contenu statique via Obsidian** : Plutôt que d’implémenter une logique deux fois, on peut choisir de la faire tourner _uniquement dans Obsidian_, puis de **publier le résultat statique**. Nous avons évoqué Dataview Publisher plus haut, mais on peut aussi recourir au plugin **Templater**. Par exemple, via un modèle Templater on peut appeler l’API Dataview (objet `dv`) pour exécuter une requête et écrire son résultat dans une note avant publication[joschua.io](https://joschua.io/posts/2023/09/01/obsidian-publish-dataview/#:~:text=Luckily%20for%20us%2C%20Dataview%20can,a%20query%20and%20returns%20Markdown)[joschua.io](https://joschua.io/posts/2023/09/01/obsidian-publish-dataview/#:~:text=%2F%2F%20Update%20Publish%20Files.md%20%3C,mtime%20desc%20LIMIT%207). C’est ce qu’a fait un utilisateur pour actualiser automatiquement la liste de ses notes récentes sur son site Obsidian Publish[joschua.io](https://joschua.io/posts/2023/09/01/obsidian-publish-dataview/#:~:text=TABLE%20WITHOUT%20ID%20file,mtime%20desc%20LIMIT%207)[joschua.io](https://joschua.io/posts/2023/09/01/obsidian-publish-dataview/#:~:text=This%20is%20because%20the%20Dataview,do%20not%20support%20dynamic%20queries). Cette approche, valable aussi pour Jekyll, consiste à _pré-calculer_ vos tableaux/listes dynamiques dans le Markdown. **Avantage** : plus besoin de coder quoi que ce soit côté Jekyll (Liquid), le Markdown a déjà la liste à jour. **Inconvénient** : il faut penser à exécuter le modèle ou le plugin avant de déployer. Des plugins comme _Dataview Publisher_ automatisent cela (mise à jour sur enregistrement, etc.[github.com](https://github.com/udus122/dataview-publisher#readme#:~:text=Image%3A%20commands_screenshot)).
    
- **Exploiter Jekyll uniquement (mode “CMS”)** : À l’inverse, vous pourriez décider de ne pas utiliser Dataview du tout, et de laisser Jekyll s’occuper de tout l’aspect base de données (tri, filtre). Par exemple, pour afficher un tableau de fiches, écrivez manuellement le code Liquid dans un fichier d’inclusion Jekyll. Ainsi, **dans Obsidian vous ne verriez pas le rendu dynamique** (juste le code Liquid ou rien du tout si mis en snippet externe), mais le site Jekyll afficherait correctement la liste. Cette solution élimine la duplication de logique, puisqu’elle n’existe plus que dans Liquid. Cependant, elle réduit l’expérience côté Obsidian (pas de prévisualisation directe des listes générées). C’est un choix à faire selon ce que vous privilégiez : l’aperçu local (alors mieux vaut générer statiquement via Obsidian) ou la simplicité du site.
    
- **Plugins de requêtes alternatives** : Il existe des plugins Obsidian permettant des requêtes de manière différente, parfois plus exportables. Par exemple, **Query all the things** permet d’exécuter des requêtes SQL sur vos notes et de formater le résultat avec Handlebars[obsidianstats.com](https://www.obsidianstats.com/plugins/bulk-exporter#:~:text=Obsidian%20notes,by%20wenlzhang%20An%20Obsidian%20plugin). On pourrait imaginer l’utiliser pour produire du Markdown standard. Ce n’est pas fondamentalement différent de Dataview en termes de duplication de logique, mais cela montre qu’il y a des _alternatives techniques_. De même, certains utilisent **CSV** : si vos données sont dans un CSV, un plugin Obsidian ou un script Python peut générer une table Markdown à partir de ce CSV, tandis que Jekyll peut aussi lire le CSV via Liquid. Ce n’est pas “un seul outil miracle”, mais une stratégie : _utiliser un format pivot simple (CSV, JSON, YAML) et deux lecteurs différents_. L’important est que le _contenu_ n’est écrit qu’une fois.
    

En somme, si une **totale mutualisation** de Dataview et Liquid n’est pas possible, on peut _tricher_ en **faisant pré-calculer les données d’un côté** (via plugin) puis en les utilisant statiquement de l’autre. Ou bien en s’appuyant sur des fichiers de données bruts partagés. Cela réduit drastiquement les duplications de code ou de données.

## Quartz, Obsidian Publish et frameworks alternatifs (avantages/inconvénients)

Enfin, il peut être utile de considérer d’autres solutions de publication qui s’accommodent mieux du contenu Obsidian, afin de voir leurs atouts et limites dans ce contexte :

- **Obsidian Publish** (service officiel) : La solution la plus simple : elle publie votre voûte Obsidian en ligne _telle quelle_. Aucun besoin de Jekyll, pas de configuration de build : il suffit de choisir les notes à publier. **Avantages** : compatibilité maximale avec les fonctionnalités Obsidian (liens internes, `![[transclusions]]` gérés nativement, vue graphe, etc., puisque c’est Obsidian qui rend le site côté client). De plus, _zéro duplication de code_ puisque vous n’écrivez aucune templating Liquid. **Inconvénients** : c’est un service payant et propriétaire. Surtout, les **plugins communautaires ne tournent pas** sur Publish, donc Dataview n’y fonctionne pas dynamiquement[joschua.io](https://joschua.io/posts/2023/09/01/obsidian-publish-dataview/#:~:text=Image%3A%20Obsidian%20Publish%20site%20where,displayed%20in%20a%20code%20block). Vous devez donc tout de même déployer des astuces (comme celles vues plus haut) pour intégrer le résultat de Dataview en statique. En bref, Publish est imbattable pour la simplicité, mais moins flexible si vous avez beaucoup de contenu dynamique ou des besoins de mise en page très custom.
    
- **Quartz** (Jackyzhao’s Quartz) : Un _framework_ de site statique spécialement conçu pour publier des notes Obsidian sous forme de **digital garden** moderne. Quartz n’est pas basé sur Jekyll (il s’agit d’un générateur Node.js, récemment en version 4). **Avantages** : support intégré des features Obsidian _out of the box_ : il gère les wikilinks, les transclusions, les backlinks, la recherche en texte intégral, une vue graphe interactive, etc.[quartz.jzhao.xyz](https://quartz.jzhao.xyz/#:~:text=Features). En gros, il apporte ce que vous auriez bricolé sous Jekyll via divers plugins/liquid, mais déjà codé dans l’outil. Quartz se veut **“batteries-included”** pour Obsidian : par exemple, `![[Note]]` marchera directement, pas besoin d’un script de conversion de liens. **Inconvénients** : c’est un environnement différent, nécessitant Node/npm et une certaine configuration. La génération du site peut être plus complexe à héberger sur GitHub Pages (souvent on passe par un déploiement sur Vercel ou on build manuellement puis on pousse le HTML). Si vous êtes déjà à l’aise avec Jekyll, Quartz implique d’apprendre un autre outil. Par ailleurs, Quartz ne prend pas en charge Dataview _tel quel_ – vous devriez tout de même exporter vos données statiquement (ou développer un plugin Quartz si possible). En résumé, Quartz est un excellent compromis pour éviter les tracas de compatibilité Obsidian/Jekyll (il _est conçu_ pour Obsidian), mais il introduit un autre écosystème technique.
    
- **Thèmes Jekyll optimisés pour Obsidian** : Il existe des templates Jekyll créés par et pour des utilisateurs Obsidian. Par exemple :
    
    - _Jekyll Garden_ (dérivé de Simply Jekyll) : thème open-source qui permet de publier un jardin numérique à partir d’une voûte Obsidian[forum.obsidian.md](https://forum.obsidian.md/t/jekyll-garden-new-jekyll-theme-for-obsidian-users/23296#:~:text=For%20the%20last%206%20months%2C,or%20see%20it%20in%20action)[forum.obsidian.md](https://forum.obsidian.md/t/jekyll-garden-new-jekyll-theme-for-obsidian-users/23296#:~:text=,Light%20theme%2C%20System%20Theme%20support). Il prend en charge les liens au format `[[...]]` (convertis en liens HTML), la page d’accueil en forme de flux, le mode sombre, etc. **Avantage** : se base sur Jekyll (compatible GitHub Pages), donc pas de serveur exotique nécessaire. Le workflow reste du **Markdown + GitHub Pages** classique, avec juste un thème adapté. **Inconvénient** : cela reste du Jekyll, donc pour des fonctions avancées comme Dataview, il faudra intégrer manuellement (via des includes ou pré-calculs). Le thème donne une bonne base pour Obsidian, mais ne supprime pas le besoin d’astuces si vous avez beaucoup de contenu dynamique.
        
    - _Digital Garden Jekyll Template_ (par Maxime Vaillancourt) : un autre starter Jekyll orienté « garden » qui facilite la connexion Obsidian→GitHub Pages. Similaire dans l’idée, il fournit des gabarits pour les backlinks, les liens wiki, etc., de sorte que vous puissiez écrire dans Obsidian sans trop vous soucier de la syntaxe finale du site[forum.obsidian.md](https://forum.obsidian.md/t/jekyll-integration/8544#:~:text=%2A%20Conversion%20from%20Zettelkasten%2Fwiki,be%20preserved%20with%20much%20attention). **Pros** : gratuit, assez documenté par la communauté. **Cons** : un projet tiers, avec possiblement moins de fonctionnalités que Quartz ou le plugin d’Ole Eskild.
        
    - D’autres (Notenote.link, Simply Jekyll, etc.) existent, chacun avec ses spécificités, mais partagent le même but : _réduire l’effort pour publier des notes Obsidian sur Jekyll_. Ils vous évitent de recoder la conversion des liens ou la mise en page de base, mais pour les choses comme Dataview, on revient aux solutions précédentes (génération statique ou duplication de logique Liquid).
        

**En résumé**, si votre préoccupation majeure est la duplication de code entre Obsidian/Dataview et Jekyll/Liquid, vous avez deux stratégies :

1. _Améliorer l’intégration dans votre stack actuelle_ (via plugins d’export, includes communs, centralisation des données, etc. comme détaillé ci-dessus) pour que votre workflow Obsidian ↔ Jekyll soit plus DRY (_Don’t Repeat Yourself_).
    
2. _Changer de stack de publication_ pour une solution qui épouse mieux Obsidian (Quartz, Obsidian Publish, etc.), au prix d’un effort technique ou financier différent, mais qui élimine certains doublons d’implémentation.
    

Chaque approche a ses avantages. Si vous tenez à Jekyll + Obsidian, les outils comme **Dataview Publisher** et l’usage malin des **includes/transclusions** vous permettront déjà de nettement réduire les redondances de code en gardant une **seule source de vérité** pour vos contenus et données[obsidianstats.com](https://www.obsidianstats.com/plugins/bulk-exporter#:~:text=The%20Bulk%20Exporter%20plugin%20allows,The%20plugin)[stackoverflow.com](https://stackoverflow.com/questions/41966638/jekyll-on-github-pages-include-markdown-in-another-markdown-file#:~:text=,md%20and%20the%20live%20page). Bon jardinage numérique !
